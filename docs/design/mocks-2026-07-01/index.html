Murmur — Rebuild UI Mocks

Static HTML/CSS mockups for the Murmur rebuild: AI meeting notes for blue-collar field work. One button starts a site walk; the on-device agent then composes a report, todos, and updated records. Not a chat app — no prompt box, no message thread. Every screen is rendered inside a 390×844 phone frame. Dated 2026-07-01.

Direction choice for screens 3–6

I built both Jobsite (dark, safety-orange) and Paper (warm off-white, burnt-orange ink) for Home and Recording, and chose Jobsite for screens 3–6. Why:

  • Tool identity. The brief asks for "DeWalt drill energy." Dark chassis + safety-orange reads as equipment; the orange record bar looks like a physical trigger. That confidence is the product's whole pitch versus SaaS note apps.
  • Affordance clarity. On a busy jobsite, one saturated orange = "the action." A single strong accent on a dark ground makes the primary control impossible to miss.
  • Honest caveat (why Paper still ships as A/B): in direct overhead sun, a dark screen mirrors glare and a light "paper" field with dark ink is often more legible. If field testing shows glare is the dominant failure, Paper is the better production choice — the two are the same components, only the palette differs, so switching is cheap. I'd validate this outdoors before locking it.

UX problems noticed while mocking

Onboarding / BYOK — the real risk

  • "API key" is a wall. A landscaper does not have, and will not create, an OpenAI/ Anthropic account from a paste field. I reframed it as "connect the brain" and added a deep-link button to the provider sign-up, but honestly a raw BYOK flow will still churn a large share of this exact demographic. Recommendation: offer a managed/hosted key with a small markup as the default path, and keep BYOK as an "advanced / bring your own" option. The mock shows BYOK because it was the assigned hard problem — but the friendliest flow may be to not require a key at all.
  • Where the key lives changes constantly. A deep link to a provider's key page breaks whenever they redesign. The "get help with this step" screenshot walkthrough will rot. This argues again for hosting the key ourselves.
  • Billing anxiety is unaddressed. "You pay pennies per report" is reassuring until the first surprise bill. Needs an in-app spend meter and a hard cap the user sets during setup — not mocked here but should exist before launch.
  • Validation can only prove format, not funds. A key can be well-formed but have no credit / hit a rate limit. The green check risks over-promising. Better: do one tiny live test call during setup and report "working" only if it actually returns.

Recording & sessions

  • The transcript is a trap. Showing live text invites the user to read and "correct" it, which fights the "don't watch this, keep working" goal. I de-emphasized and masked it, but it may be worth hiding entirely behind a tap — presence of text implies you're meant to monitor it.
  • Accidental double-tap / pocket stops. One button to start and stop means a bump can end a session. Stop needs either a deliberate target (done here — full-width, distinct red) or a confirm; I avoided a confirm because it adds friction, but this needs field testing.
  • Phone-away during processing. If the user locks the phone mid-processing, on-device transcription + LLM must survive backgrounding (iOS will suspend). The offline screen promises "finishes later" — that promise requires real background/resume handling, or it's a lie.
  • Session labeling. The mock shows a pre-filled label ("Site walk — Johnson"). In reality the agent has to infer project/label after the fact, or the user must pick before recording (friction, and they won't). Likely answer: record first, let the agent guess the project from content, let the user correct on the report.

Home & agent composition

  • "Needs your call" must be rare. The alert banner is powerful precisely because it's uncommon. If the agent surfaces a decision every session, it becomes noise and the whole "only what matters" promise collapses. Needs a strict budget (≤1 alert, and only for genuine money/schedule decisions).
  • Trust in extraction. Auto-created todos/contacts with wrong assignees or hallucinated people erode trust fast for this user. Edit affordances are everywhere in the report, but the agent should lean toward under-extracting (fewer, confident items) over dumping everything.
